# Liquers Comprehensive Project Review — 2026-07-07

## 1. Executive Summary

Liquers is in a healthy but transitional state. The build is green across the whole workspace
(including `liquers-py`), and **~700 tests pass with zero failures** (300 unit tests in
`liquers-core`, 263 in `liquers-lib`, plus integration tests for dependencies, expiration,
volatility, UI, Polars and stores). The recent work (Jan–Mar 2026) focused on the
dependency/expiration system and has left a well-documented trail of design documents
(`specs/ASSET_LIFECYCLE.md`, `specs/metadata-consistency/FINDINGS.md`, `specs/FEATURES/*`).

The dominant risk concentration is in **`liquers-core/src/assets.rs` (4,866 lines)** — the asset
lifecycle, scheduler and expiration monitor. This single module is the enabler for everything else
(HTTP API, UI, Python bindings, dependency tracking all sit on top of it), and it contains the only
confirmed *hang-class* bug in the project (recipe delegation deadlock), several race-condition
classes acknowledged in draft specs, and most of the technical debt (debug `println!`s, duplicated
evaluation paths, an unfinished message-lifecycle contract).

**Recommended focus order** (detailed in §5):
1. Make asset execution *unable to hang* (scheduler/delegation deadlock).
2. Make asset failure *unambiguous* (error/notification/metadata contract).
3. Make expiration *race-free* (stale reads, monitor consistency).
4. Make stored data *self-consistent* (metadata `data_format` contract, OpenDAL path bugs).
5. Only then invest in the larger refactors (ValueInterface split, Context slimming) and new
   features (openbin, combined expires, extended fast-track, session/ACL).

The rationale for this ordering: items 1–4 are **correctness of the foundation layer**; every
planned feature (web API, UI, workflow engine `liquers-wf`, Python integration) calls through the
asset layer, so fixing it first maximizes leverage and avoids re-testing the stack repeatedly.
Refactors (ValueInterface, Context) are intentionally scheduled *after* correctness fixes, because
refactoring on top of racy behavior makes regressions undetectable — and the planned
`BENCHMARK-SUITE` should land before performance-motivated refactors so improvements are measurable.

## 3. Findings

Findings are labeled F-1…F-20 and referenced by the priority plan in §5.

### 3.1 Critical correctness (hang / data-loss class)

#### F-1: Recipe delegation can deadlock the job queue — CONFIRMED
`AssetRef::evaluate_recipe()` delegates pure-key recipes to another asset via
`return asset.get().await` (`liquers-core/src/assets.rs:1316`, explicit `FIXME: !!!`), while the
delegating asset **still occupies a job-queue slot**. `JobQueue::submit()` queues jobs as
`Submitted` when `running_count >= capacity` (capacity is hardcoded to 4 in
`DefaultAssetManager::new()`, `assets.rs:2204`). With a delegation chain deeper than the queue
capacity, children never get a slot and the parent waits forever in `get()` — a guaranteed hang,
no timeout, no diagnostic. Tracked as `CORE-ASSETS-RECIPE-DELEGATION-DEADLOCK` and promoted to
`SCHEDULER-IMPROVEMENTS`/`ASSETS-FIX1`, but still unfixed.

**Why top priority**: it is the only known input-dependent *hang* in the core execution path; any
user composing recipes that reference other keyed recipes can trigger it. It also blocks the
`Status::Dependencies` design (`specs/DEPENDENCIES_STATUS.md`), which is the intended cure.
The fix direction in `todo20260219.md` §3 (non-blocking dependency scheduling: mark parent
`Dependencies`, release the slot, resume event-driven) is right and should be implemented as
specified, together with cycle detection and the capacity=1 regression tests.

#### F-2: Asset failure semantics are ambiguous (three different outcomes for one failure)
Combination of confirmed code paths:
- `AssetData::poll_state()` returns `Some(State)` with a `none` value for `Status::Error |
  Cancelled` (`assets.rs:604`), so `AssetRef::get()` returns **`Ok(error-flavored state)`** if it
  polls after the status is set.
- If the waiter instead wakes on the `watch` notification channel, it may see
  `ErrorOccurred(e)` → returns **`Err(e)`**.
- Because `watch` keeps only the latest message, `ErrorOccurred` can be overwritten by the
  subsequent `JobFinished` before the waiter reads it; `get()` then returns
  **`Err("Asset finished but no data available")`** — the real error is lost (`assets.rs:1821`).

So the same failed evaluation yields `Ok(..)`, `Err(real error)`, or `Err(generic)` depending on
race timing. This directly hits the HTTP API, the UI's `AssetViewElement` and Python bindings.
It is the concrete manifestation of the one open issue in `specs/ISSUES.md`
(`ASSET-MESSAGE-LIFECYCLE-ROBUSTNESS`, priority High).

**Fix direction**: make asset status + stored error the single source of truth; `get()` should
consult status (not the last notification) for terminal outcomes, and the notification channel
should be treated as a wake-up signal only. Define the post-finish message policy as the issue
requests, and add the late-message/race tests it lists.

#### F-3: Error path wipes metadata history
`finish_run_with_result()` does `lock.metadata = Metadata::from_error(e.clone())`
(`assets.rs:1184`) — replacing the entire metadata, discarding the log, query information, type
info and any recorded dependencies. For a framework whose selling point includes "complete audit
trail of asset creation" (PROJECT_OVERVIEW), failures are exactly where the trail matters most.
Related: `CORE-METADATA-TRACEBACK-SUPPORT` (no error-chain support in metadata).

**Fix**: add the error *into* the existing `MetadataRecord` (`with_error` already exists and is
used in `evaluate_and_store`'s error branch — the two error paths should be unified anyway,
see F-9).

#### F-4: Expiration races and stale reads — partially fixed, remainder designed but unimplemented
Recent commits (Jan–Mar 2026) fixed the triple-expiration-setting redundancy and the volatile
overwrite bug (ASSET_LIFECYCLE Issues 1–2, marked FIXED). What remains, per
`specs/FEATURES/EXPIRATION-SAFETY.md` (Draft) and confirmed in code:
- `poll_state()` returns data for `Status::Expired` assets (`assets.rs:612`) — an expired asset
  serves stale data to `get()` with no revalidation guard.
- The expiration monitor keeps **strong `AssetRef`s** in its heap (`TimedAsset`,
  `assets.rs:2140`); `Untrack` only removes the id from the deadline map, leaving the heap entry
  (and the asset it keeps alive) until the timer fires — unbounded retention for far-future
  deadlines and re-tracked assets.
- Duplicate Track/fire logic exists in the two `select!` arms of `run_expiration_monitor`
  (`assets.rs:2252-2384`) — a maintenance hazard where a fix to one arm can miss the other.

**Why P0-adjacent**: the whole dependency/cascade-expiration system built this spring feeds into
these paths; races here silently undermine it. The implementation plan already exists
(`EXPIRATION-SAFETY-IMPLEMENTATION-PLAN.md`) — execute it.

### 3.2 High-impact correctness (wrong results, real backends)

#### F-5: Metadata format contract is not enforced (silent read/write asymmetry)
`specs/metadata-consistency/FINDINGS.md` is thorough and correct; the key confirmed risks:
serialization uses `metadata.get_data_format()` while at least one deserialization path keys off
the filename extension; `set()`/`set_state()` accept arbitrary metadata although
`ASSET_SET_OPERATION.md` mandates `data_format`/`type_identifier`; `State` public fields allow
constructing inconsistent states (4 different creation patterns inventoried). This is a
*silent-corruption* class bug for anything persisted, and it blocks safe multi-client use
(HTTP API and Python write into the same stores).

**Fix**: implement the candidate invariants from FINDINGS.md — decide the strictness policy
(recommend: normalize-and-warn first, strict later behind a flag), enforce effective
`data_format` on the read path, and funnel `State` creation through checked constructors.

#### F-6: OpenDAL store slash/path normalization bugs
`STORE-OPENDAL-SLASH-HANDLING-BUG` (todo audit §6, verified there in code): `listdir` vs
`listdir_keys_deep` use different path conventions, `get_metadata()` has a `FIXME` disabling
directory children, roundtrips are backend-dependent, and `test_opendal_subdir` deliberately
avoids strict assertions. This means **directory listing is unreliable on exactly the backends
OpenDAL exists to enable** (S3 & friends).

**Fix**: centralize key↔path mapping in one module with the roundtrip property
`path_to_key(key_to_path(k)) == k`, then re-enable strict tests. Medium effort, high payoff,
independent of the asset work — good parallel track.

#### F-7: JobQueue is a busy-wait loop with no shutdown and no backpressure
`JobQueue::run()` polls every 100 ms forever (`assets.rs:3428-3503`), scans the whole job list
under a mutex (awaiting per-asset status *while holding the lock*), holds strong refs to all
jobs, and has no shutdown path (the expiration monitor at least has `Shutdown`). Capacity is
hardcoded (4). This is not a data-corruption bug, but it is a design flaw in the exact component
that must be reworked for F-1 — so fold it into the scheduler rework rather than patching twice:
event-driven wakeup (notify on submit/finish), configurable capacity, graceful shutdown,
weak refs or explicit removal.

#### F-8: Plan builder gaps: `expand_predecessors` crash and undocumented policies
`Recipe::to_plan` carries a commented-out `disable_expand_predecessors()` because it *crashes a
unit test* (`recipes.rs:157`), and plan.rs has TODO'd cache/volatile/inline policy defaults
(todo audit §7, §8, §15). Relative key/query resolution in dependency discovery is also TODO
(`plan.rs:1533`, `plan.rs:1668`) — meaning dependency tracking of relative references is
incomplete, which quietly weakens the new dependency-management system.

**Fix**: regression test for the crash; introduce `PlanBuilderConfig` with documented defaults;
implement canonical cwd-aware resolution (this last part is HARD and is the true enabler for
correct recipe-relative dependencies).

### 3.3 Design flaws (architecture level)

#### F-9: Duplicated, asymmetric evaluation paths
`evaluate_and_store` vs `evaluate_immediately` diverge exactly as documented in
`ASSET_LIFECYCLE.md` §6: the immediate path never persists, never registers in the
DependencyManager, and **drops runtime dependencies** (`take_pending_dependencies` never called
— Issue 3), while `resolve_volatility_before_evaluation()` is called redundantly up to 4× per
run (Issue 6). `Status::Dependencies` is effectively dead code today (Issue 4) but is the
keystone of the F-1 fix — implement it there rather than deleting it.
This cluster is `ASSETS-FIX1` Phases 2–3; the analysis is done, the consolidation just needs to
be executed *after* F-1/F-2 define the target semantics.

#### F-10: `ValueInterface` is a monolith coupled to serialization
`CORE-VALUE-INTERFACE-ARCHITECTURE-DEBT` (todo audit §9): one trait bundles construction,
extraction, type identity, JSON, and (via supertrait) binary serialization. Every new value type
in `liquers-lib` and every Python-exposed type pays the full contract. The audit's
Alternative A (capability-trait decomposition with a blanket compatibility trait) is the right
call — lower migration risk. **Do it after F-5**, since the metadata/type-identifier invariants
will dictate part of the new trait boundaries (`type_identifier` vs `type_name` split).

#### F-11: `Error` enum is too large by value
Clippy reports **421×** "the `Err`-variant returned from this function is very large". `Error`
(with position, query, key context) is returned by value on virtually every function in the
workspace, including the performance-sensitive parse/key paths named in CLAUDE.md. Boxing the
payload (`struct Error(Box<ErrorInner>)`) is a mostly mechanical, crate-local change that
removes ~70% of all clippy noise and shrinks every `Result` in the codebase.

#### F-12: No session/authorization model
`Session` is a stub (`CORE-CONTEXT-SESSION-MODEL-TODO`), and `KEY-LEVEL-ACL` (write-side
authorization for `set`/`set_state`) is an open feature. PROJECT_OVERVIEW correctly notes the
hard part: assets are shared across users, so authorization must live at access points. **This
is a prerequisite for any non-trusted deployment of `liquers-axum`** — currently the HTTP API
must be treated as trusted-network only. It doesn't block local/desktop use, hence P2, but it
gates the web story and should be designed before the web API grows more surface.

#### F-13: Context has accumulated asset-lifecycle responsibilities
`Context` forwards `set_value/set_state/set_error/set_expires` to `AssetRef` and owns
`pending_dependencies` (ASSET_LIFECYCLE Issue 7). A custom `Environment::apply_recipe` that
forgets `context.set_expires()` silently loses expiration info (documented trap). Slim Context
to an execution facade per the existing recommendation; fold into the F-9 consolidation.

### 3.4 Code quality / hygiene

#### F-14: 183 `println!`/`eprintln!` in `liquers-core` production code
`assets.rs` alone has 60 (e.g., every `get()` loop iteration prints). For a library embedded in
servers, GUIs (egui!) and Python, stdout noise is a real defect, and several prints leak
internal state. Replace with `tracing` (levels + targets); this also serves the observability
needs of F-1/F-2 debugging. Quick, mechanical, high visible payoff.

#### F-15: CLAUDE.md constraint violations: `unwrap/expect` in library code
Examples outside tests: `DefaultAssetManager::get_envref()` uses `.expect("Environment not set…")`
(`assets.rs:2396`) — reachable panic during misordered initialization; ~56 occurrences in
`expiration.rs`, ~89 in `plan.rs` (some in tests; needs triage). Convert to `Error` returns or
documented invariants. Also 4 clippy "unused `Result` that must be used" hits deserve a look —
ignored `Result`s in async code are how errors vanish.

#### F-16: Dead/legacy modules and uncommitted work
`entities.rs` is 0 lines (delete). `cache.rs` (342 lines) is legacy from the Python design and
PROJECT_OVERVIEW already says assets provide natural caching — decide (remove or document), don't
let it bit-rot. Root-level scratch files (`example1_cascade_expiration.rs`, `EXAMPLE1_SUMMARY.md`,
`Untitled.ipynb`) should be committed to a proper location or removed; several months of
untracked spec work (`specs/ASSET_LIFECYCLE.md`, `specs/value-accessor/`, `specs/liquers-wf/`)
is sitting uncommitted in the working tree — commit it, it's good material. Separately: the
`target/` directory had grown to 65 GB and filled the disk during this review (incremental cache
was purged); consider `cargo clean` periodically or a CI-side build.

### 3.5 Missing features (planned, not yet started)

#### F-17: `openbin` streaming (CORE-STORE-OPENBIN-MISSING)
Unimplemented across store layers (`store.rs:207` and 4 more sites). Matters for large binary
assets (images, Parquet). Medium effort, unblocks large-data workflows; sequence after F-6 so it
lands on sane path handling.

#### F-18: Macro validation & argument hints (MACRO-QUERY-VALIDATION-AND-HINTS)
`query "..."` literals unvalidated at compile time; `hint` not propagated into
`ArgumentInfo.hints`. Cheap developer-experience win; independent of everything else.

#### F-19: Planned feature backlog (queue behind correctness work)
`listdir_keys`/`listdir_keys_deep` HTTP routes missing (audit item 17, partially open); WebSocket
asset notifications exist but their robustness depends on F-2's message contract. Also
`COMBINED-EXPIRES` (expiration algebra), `EXTENDED-FAST-TRACK` (command execution classes),
`VALUE-DESCRIPTION`, `COMMAND-METADATA-ENHANCEMENTS`, `COMBINED-VALUE-DISCRIMINATION`,
`EGUI-ASSET-MANAGER-INTEGRATION` — all reasonable, none blocked externally.

#### F-20: Strategic gaps acknowledged in PROJECT_OVERVIEW
Multi-realm interpreter (designed, unimplemented), asset garbage collection (not designed —
note F-4/F-7 currently *retain* assets via strong refs, so GC design should cover those),
dependency checking completion, and user documentation. Docs deserve a slot before any public
release: the Python docs are declared obsolete and there is no current user-facing documentation.

## 7. Closing Notes

- The project's own issue tracking is unusually good; this review largely **confirms and
  sequences** existing knowledge, adding a handful of newly verified specifics: the
  `Ok`-vs-`Err` ambiguity of `get()` on failed assets (F-2), the metadata wipe on the error path
  (F-3), stale-data serving for `Expired` status via `poll_state` (F-4), strong-ref retention in
  the expiration monitor heap (F-4), the busy-wait/no-shutdown JobQueue characteristics (F-7),
  and the clippy-quantified `Error`-size problem (F-11).
- The single most valuable near-term milestone: **"an asset evaluation can always terminate, and
  its outcome is always unambiguous."** Steps 1–3 of the plan deliver exactly that, and every
  other roadmap item (web API, workflow engine, Python, UI) inherits the benefit.
