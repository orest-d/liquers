### 3. CORE-ASSETS-RECIPE-DELEGATION-DEADLOCK (HARD)
Description:
`AssetRef::evaluate_recipe()` delegates pure-key recipes to another asset via `return asset.get().await`.
This call happens while the current asset is already in `Processing` state and still occupies a job-queue slot.
If delegation depth exceeds available queue capacity, the delegated asset can remain only `Submitted`, while the parent waits indefinitely in `get()`.
This is explicitly warned in code (`FIXME` near the delegation path).

Verification (2026-02-20):
Confirmed by code-path inspection:
1. Delegation branch: `evaluate_recipe()` -> `asset.get().await` (`liquers-core/src/assets.rs:1045`-`1072`).
2. `get()` waits for completion notifications and can hang if job never starts (`liquers-core/src/assets.rs:1441`-`1487`).
3. `JobQueue::submit()` sets `Submitted` when `running_count >= capacity` (`liquers-core/src/assets.rs:2370`-`2423`).
4. Therefore with chain length `N` and queue capacity `C`, if `N > C`, a waiting parent can block children from ever getting a slot.

Potential solution:
1. Replace synchronous delegation wait with non-blocking dependency scheduling:
   1. In delegation branch, submit/ensure delegated asset is scheduled, then return a dedicated status (e.g. `Dependencies`) instead of waiting in `get()`.
   2. Let queue worker re-check dependency readiness and resume parent when child reaches terminal state.
2. Add explicit dependency edge tracking (`parent_asset_id -> child_asset_id`) so parent state transitions are event-driven, not blocking.
3. Add cycle and self-delegation guards:
   1. Existing self-id check stays.
   2. Reject/mark error on detected delegation cycles (A->B->A) with clear metadata log.
4. Add regression tests:
   1. Capacity=1 and 2-level delegation should complete.
   2. Capacity=2 and 3-level delegation should complete.
   3. Delegation cycle should fail fast with `Error`, not hang.
5. Optional safety net: bounded timeout/warning in `get()` for long-lived `Submitted/Dependencies` states to surface scheduler stalls.

### 4. CORE-ASSETS-PERSISTENCE-GAPS (HARD)
Description:
Asset mutation paths do not share one persistence contract. Some code paths write value+metadata durably to store, while others only update in-memory state.
This can lead to runtime-visible state that is not recoverable after restart, and to metadata drift across APIs.

Verified gaps (2026-02-20):
1. `AssetRef::set_value` (`liquers-core/src/assets.rs:1542`) updates in-memory `data/status/metadata` and emits notifications, but does not persist to store (`TODO: Store value in set_value`).
2. `AssetRef::set_state` (`liquers-core/src/assets.rs:1577`) similarly updates in-memory state, but has `TODO: Store state in store`.
3. `set_state` has an internal metadata ordering problem:
   1. it writes inferred `type_identifier/type_name` to `lock.metadata`,
   2. then immediately replaces `lock.metadata` with `state.metadata` (`liquers-core/src/assets.rs:1584`-`1589`), effectively discarding those inferred type updates.
4. These non-persistent setters are reachable from normal command execution context (`liquers-core/src/context.rs:305`, `liquers-core/src/context.rs:315`), so this is not dead code.
5. Fast-track coverage is incomplete for non-resource assets (`TODO: support for quick plans`, `liquers-core/src/assets.rs:372`), increasing behavioral inconsistency across evaluation paths.
6. Recipe evaluation can produce a value visible via `asset.get()` before durable write succeeds:
   1. `evaluate_and_store` emits `ValueProduced` before persistence path completes (`liquers-core/src/assets.rs:1148`-`1173`),
   2. default mode is background save (`save_in_background = true`, `liquers-core/src/assets.rs:279`),
   3. therefore callers can observe ready value while store write is still pending.
7. Save failures are currently swallowed in evaluation flow:
   1. background path ignores `save_to_store` result (`let _ = assetref.save_to_store().await`, `liquers-core/src/assets.rs:1168`),
   2. foreground path also ignores `save_to_store` result (`let _ = self.save_to_store().await`, `liquers-core/src/assets.rs:1171`).
8. Missing store key is a valid failure mode that can be silently lost:
   1. `save_to_store` fails when `recipe.store_to_key()` is `None` (`liquers-core/src/assets.rs:1238`-`1247`),
   2. this failure is currently not propagated to callers due to ignored `Result`.

Scope clarification:
`AssetManager::set` and `AssetManager::set_state` already perform store writes with validation/normalization (`liquers-core/src/assets.rs:2038`, `liquers-core/src/assets.rs:2148`), but internal `AssetRef` mutation paths still bypass equivalent durability rules.

Proposed solution:
1. Define a single write-through contract for asset mutations:
   1. normalize metadata,
   2. set final status,
   3. serialize/store (or metadata-only fallback for non-serializable values),
   4. emit notifications.
2. Refactor `AssetRef::set_value` and `AssetRef::set_state` to call a shared internal persistence routine used by `save_to_store`/manager setters.
3. Fix metadata assignment order in `set_state`: merge state metadata first, then enforce inferred `type_identifier/type_name`.
4. Add tests for consistency across update APIs:
   1. `Context::set_value` survives reload from store,
   2. `Context::set_state` persists metadata/status correctly,
   3. inferred type fields are present after `set_state`.
5. Make persistence outcome observable:
   1. do not discard `save_to_store` errors; propagate or record them in metadata/log with an explicit persistence status,
   2. if background save is kept, add notification/state for `Persisted` vs `ReadyButNotPersisted`.
6. Add dedicated tests for observed behavior:
   1. default background mode: `asset.get()` may complete before store contains key (documented/expected) or enforce opposite and test it,
   2. simulated store write failure must be visible to caller/metadata (not silent),
   3. recipe with no storable key must produce deterministic and test-covered outcome.
7. Decide and document quick-plan fast-track behavior (either implement or explicitly disable with rationale) to remove path-dependent persistence semantics.

Open decisions needed:
1. Should `asset.get()` mean "value computed" only, or "value computed and durably persisted" when recipe has `store_to_key()`?
2. For background save failures, preferred behavior:
   1. fail `get()`/evaluation immediately,
   2. keep value available but mark metadata with persistence error and warning,
   3. configurable policy.
3. For recipes without storable key, should this be:
   1. hard error at evaluation time,
   2. allowed in-memory-only result with explicit metadata flag/log.

### 6. STORE-OPENDAL-SLASH-HANDLING-BUG (MEDIUM)
Description:
OpenDAL adapter has inconsistent key<->path normalization around `'/'` (leading/trailing separators and directory prefixes), causing incorrect directory/listing behavior.
Root cause note: behavior differs depending on whether directory paths are queried with a trailing slash.

Verified in code (2026-02-20):
1. `listdir()` forces trailing slash (`...trim_end_matches('/') + "/"`) but `listdir_keys_deep()` does not (`key_to_path(key)` directly), so the same logical key can be listed with different path semantics (`liquers-store/src/opendal_store.rs:424`, `liquers-store/src/opendal_store.rs:455`).
2. `path_to_key()` globally trims both leading and trailing slashes (`trim_matches('/')`), which can collapse directory identity at boundaries and makes roundtrip path handling backend-dependent (`liquers-store/src/opendal_store.rs:236`).
3. Directory child names can come back as full prefixed paths instead of local names, then re-joined via `key.join(name)`, producing wrong keys for nested dirs in some cases (`liquers-store/src/opendal_store.rs:430`-`447`).
4. `get_metadata()` explicitly disables directory children population due to this slash/path issue (`FIXME` in `liquers-store/src/opendal_store.rs:312`-`314`).

Situations where it appears:
1. Listing nested directory content (`listdir("sub")`, `listdir_keys("sub")`) on backends that return entry names/paths with parent prefix or trailing `/`.
2. Building deep key trees (`listdir_keys_deep`) where recursive listing returns mixed directory/file path formats.
3. Metadata read for directory keys when metadata file is absent: `get_metadata(dir)` cannot safely populate `children`.
4. Root and near-root operations (`Key::new()`, single-segment keys) when one path branch uses explicit `/` suffix and another does not.
5. Backend-specific behavior differences (memory vs filesystem/object-store backends), because OpenDAL entry path conventions vary.
6. Same logical directory queried as `path` vs `path/` can produce different results in `exists/list/stat` on some OpenDAL backends.

Observed test hint:
`test_opendal_subdir` already avoids strict assertions for nested list output, indicating unstable/incorrect path normalization in subdir cases (`liquers-store/src/opendal_store.rs:624`-`630`).

Proposed solution:
1. Centralize key<->path normalization and escaping.
2. Replace ad-hoc path manipulation in OpenDAL adapter.
3. Add tests for:
   1. root key and empty-prefix behavior,
   2. nested dirs with/without trailing slash,
   3. `listdir` name semantics (local names only),
   4. `listdir_keys` and `listdir_keys_deep` parity/invariants,
   5. `path_to_key(key_to_path(k)) == k` roundtrip for representative keys.

### 9. CORE-VALUE-INTERFACE-ARCHITECTURE-DEBT (HARD)
Description:
`ValueInterface` currently combines too many responsibilities in one trait (`liquers-core/src/value.rs:39`), while serialization is split into a second trait (`DefaultValueSerializer`, `liquers-core/src/value.rs:749`) with overlapping concerns.

Existing split of responsibilities today:
1. Value construction/factories:
   1. `none`, `new`, `from_*` for scalar/domain values (`liquers-core/src/value.rs:43`-`120`).
2. Typed extraction/conversion:
   1. `try_into_*` for scalars, bytes, key/query (`liquers-core/src/value.rs:123`-`169`).
3. Type identity and default representation metadata:
   1. `identifier`, `type_name`, `default_extension`, `default_data_format`, `default_filename`, `default_media_type` (`liquers-core/src/value.rs:170`-`199`).
4. JSON conversion and structural composition:
   1. `try_into_json_value`, `try_from_json_value`, `from_array`, `from_object` (`liquers-core/src/value.rs:200`-`251`).
5. Binary serialization/deserialization (separate trait, but bound into ValueInterface):
   1. `as_bytes`, `deserialize_from_bytes` (`liquers-core/src/value.rs:750`-`755`).
6. Trait coupling:
   1. `ValueInterface` requires `DefaultValueSerializer` in its supertraits, tightly coupling runtime value semantics with serialization implementation choices (`liquers-core/src/value.rs:39`).

Why this is debt:
1. Implementors must satisfy a very broad contract even for use cases that only need a subset.
2. Naming overlap/confusion remains (`identifier` vs `type_name` TODO notes).
3. Changes in one concern (e.g., wire formats) force touching the core value trait and all implementors.

Alternative A: Capability-Trait Decomposition (recommended)
1. `ValueTypeInfo`:
   1. responsibility: type identity + default format/media metadata.
   2. methods: `type_identifier`, `type_name`, default extension/data_format/media_type/filename.
2. `ValueConstruct`:
   1. responsibility: constructors/factories from primitive/domain inputs.
   2. methods: `none`, `from_string`, `from_i64`, `from_key`, etc.
3. `ValueExtract`:
   1. responsibility: typed extraction/conversions from value.
   2. methods: `try_into_*`.
4. `ValueJson`:
   1. responsibility: JSON roundtrip/composition.
   2. methods: `try_into_json_value`, `try_from_json_value`, `from_array`, `from_object`.
5. `ValueBinaryCodec`:
   1. responsibility: binary format serialization/deserialization.
   2. methods: `as_bytes`, `deserialize_from_bytes`.
6. Compatibility adapter:
   1. keep legacy `ValueInterface` as blanket trait/alias over these smaller traits during migration.

Alternative B: Core Value + External Codec Registry
1. `CoreValue`:
   1. responsibility: in-memory semantics only (none/check, type info, basic construction/extraction).
2. `ValueCodec` traits (external):
   1. `JsonCodec<T>` for JSON conversion.
   2. `BinaryCodec<T>` for data-format-specific byte conversion.
3. Registry layer:
   1. resolve codec by `type_identifier` + `data_format` at runtime.
   2. enables per-format/per-type codec replacement without modifying value type trait.
4. Tradeoff:
   1. more flexible for plugins and user-defined values,
   2. larger architectural change (registry lifecycle + error/reporting path updates).

Proposed direction:
1. Start with Alternative A (lower migration risk, minimal architectural shock).
2. Use compatibility adapters so command/state/assets code can migrate incrementally.
3. Re-evaluate Alternative B later if plugin codec ecosystem becomes a priority.

### 27. CORE-STORE-DOUBLE-POINTER-INDIRECTION (MEDIUM)

1. Change `Environment::get_async_store()` return type to `Arc<dyn AsyncStore>`.
2. Update all Environment implementations (SimpleEnvironment, DefaultEnvironment) to return Arc directly.
3. Remove Box wrapper from store field in environment structs.
4. Verify all store access patterns still work (should be transparent change).
