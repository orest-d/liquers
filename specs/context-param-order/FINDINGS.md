# Issue 13 Findings: CONTEXT-PARAM-ORDER

## Scope Investigated

- `specs/ISSUES.md` (issue definition)
- `liquers-macro/src/lib.rs` (macro parser + code generation)
- `liquers-core/src/commands.rs` (runtime argument retrieval)
- `liquers-core/src/command_metadata.rs` (metadata model behavior relevant to indexing)

## Problem Statement (Confirmed)

Issue 13 is valid: parameter extraction indexes generated by `register_command!` can be shifted when `context` appears before regular/injected command parameters.

Root cause is a mismatch between two macro paths:

1. Extraction code generation indexes all signature parameters, including `Context`.
2. Metadata argument list includes only actual command arguments (`Param`), excluding `Context`.

As a result, extraction may request argument `i=1` while metadata and runtime values store that parameter at `i=0`.

## Evidence

### 1. `Context` is represented as a non-argument parameter in macro AST

- `CommandParameter::Context` variant exists in `liquers-macro/src/lib.rs:457`.
- `context`/`Context` are parsed as this variant in `liquers-macro/src/lib.rs:1430`.

### 2. Extraction index currently comes from `enumerate()` over all parameters

In `CommandSignature::extract_all_parameters`:

- `self.parameters.iter().enumerate().map(|(i, p)| p.parameter_extractor(i))`
- See `liquers-macro/src/lib.rs:1003` and `liquers-macro/src/lib.rs:1005`.

This includes `CommandParameter::Context` in the index progression.

### 3. Metadata arguments intentionally exclude `Context`

- `argument_info_expression()` returns `None` for non-`Param` variants, including `Context`; see `liquers-macro/src/lib.rs:725`.
- Metadata argument vector uses `.filter_map(|p| p.argument_info_expression())`; see `liquers-macro/src/lib.rs:1221`.

So metadata compacts indices over `Param` only.

### 4. Runtime retrieval is positional and strict

- `CommandArguments::get` and `CommandArguments::get_injected` fetch by index `i`; see `liquers-core/src/commands.rs:97` and `liquers-core/src/commands.rs:130`.
- They validate expected injected/non-injected kind against the parameter at that index (`liquers-core/src/commands.rs:120` and `liquers-core/src/commands.rs:137`).

Therefore, any index shift causes wrong parameter lookup and/or type-kind mismatch.

## Concrete Failure Scenario

Signature:

```rust
register_command!(cr, fn remove(state, context, target_word: String) -> result)
```

Current extraction generation logic can assign `target_word` index `1` because `context` occupies index slot `0` during `enumerate()`.

But metadata/action parameter list has `target_word` at index `0` (because `context` is filtered out). Runtime lookup at index `1` fails or points to the wrong argument.

## Additional Notes

- Current workaround in issue text (`context` last) is correct but fragile and undocumented in enforced syntax.
- Existing macro unit tests cover extraction and wrapper generation, but there is no explicit regression test for `context` in middle position.

## Impact

- Affects correctness of command invocation for valid-looking DSL signatures.
- Can manifest as missing argument, conversion, or injected/value consistency errors at runtime.
- Most visible in commands mixing `context` with regular/injected parameters in non-tail position.

## Parameter Kind Comparison

The issue is specifically about `context`, not `injected`. `injected` arguments are treated differently and remain index-aligned with metadata.

| Dimension | Normal parameter (`name: T`) | Injected parameter (`name: T injected`) | Context parameter (`context`) |
|---|---|---|---|
| Macro AST variant | `CommandParameter::Param` | `CommandParameter::Param` with `injected=true` | `CommandParameter::Context` |
| Present in `CommandMetadata.arguments` | Yes | Yes (`ArgumentInfo.injected=true`) | No |
| Consumes query/action parameter token | Yes (via `pop_value` from iterator) | No (`pop_value` returns immediately when injected) | No |
| Present in `ResolvedParameterValues` | Yes | Yes, as `ParameterValue::Injected(name)` placeholder | No |
| Extracted in wrapper from `arguments` | Yes, via `arguments.get(i, name)` or `get_value` | Yes, via `arguments.get_injected(i, name, context)` | No extraction call; wrapper passes `context` directly |
| Value source at execution | Query/default/override/link/enum | Runtime context via `InjectedFromContext::from_context` | Wrapper `context` argument |
| Can be overridden in `ResolvedParameterValues::override_*` | Yes | No (override returns `false` for injected slots) | Not applicable |
| Participates in index space used by wrapper `arguments.get*` | Yes | Yes | No |

### Evidence for injected treatment

- Macro marks injected in metadata: `ArgumentInfo { injected: #injected, ... }` (`liquers-macro/src/lib.rs:719`).
- Planner does not consume action token for injected: early return in `ParameterValue::pop_value` when `arginfo.injected` (`liquers-core/src/plan.rs:569`).
- Planner still stores injected slot: `ParameterValue::Injected(name)` (`liquers-core/src/plan.rs:360` and `liquers-core/src/plan.rs:736` docs).
- Executor uses positional slot and validates injected kind:
  - normal: `CommandArguments::get` rejects injected (`liquers-core/src/commands.rs:120`)
  - injected: `CommandArguments::get_injected` requires injected (`liquers-core/src/commands.rs:137`)

### Clarification

Injected parameters are not query-sourced, but they are still part of the command argument vector/indexing used by metadata, planning, and wrapper extraction. `context` is the only signature item that is fully outside that argument vector.
