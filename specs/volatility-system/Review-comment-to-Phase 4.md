Review comment to Phase 4:
is_volatile should also be implemented in
- Metadata (supporting legacy metadata, defaulting to false if neither the field nor status is available). Rationale: cases with legacy or no metadata should be detected in the future and marked as either expired or override by the user. Such operation is however out of scope now.
- AssetInfo should also have is_volatile flag.
- Make sure that the links in the ResolvedParameterValues are also checked for volatility.
- A caviat to test in the query: a query "action/v" would be volatile, but "action/v/q" is not volatile, since it evaluates to Query("action/v"). If queries are checked via a plan, this would automatically be correct, but a unittest is needed.
- Approach 1 in "Step 10: Implement Phase 1 Volatility Detection in PlanBuilder" is incorrect - segment can't be an instruction, There is no is_instruction. Approach 2 however looks OK. It is highlighted that "It should NOT be passed to `process_action_request`" - that would not necessarily be a problem as long as the process_action_request updates the is_volatile and does not generate a Step other than Step::Info. It would be good to redesign the PlanBuilder to have a cleaner structure and explicitly handle instructions (is_instruction), but let's keep it for later.
- In find_dependencies, these steps will not create circular dependencies:
  * UseKeyValue - creates a value with the key, but does not on its own make an attempt to get the resource with the key, thus it does not create a dependency on the resource with the key
  * GetAssetRecipe - asset recipe is associated with the key, but it is a separate resource. In a dependency tree it is a leaf. Recipe does not have further dependencies, therefore it is not important for the circularity check. Keep this explanation in the documentation.
- - In find_dependencies, these Steps have ambiguous meaning in terms of dependencies:
  * GetResource - fetches data directly from the store. Using the store rather than assets is a mechanism bipassing dependency controls. It is OK for now to treat it as no dependency, but flag it in the doc string.
  * SetCwd - on its own does not create the dependency, but it impacts relative links. This needs a more complex evaluation: when a key or query is examined for circular dependency, it must try to find valid Cwd (previous SetCwd step in the plan), expand the query to its absolute form and then assess the expanded query/key. The find_dependencies should have an Optional cwd parameter.
- find_dependencies if it finds a circular rependency, the error should say what (which key) caused the circular dependency. The key should be set on the Error obbject (with_key?)
- As a rule, avoid default match branches (e.g. in find_dependencies)
- regarding the make_plan - On some places the plan can be made with the PlanBuilder bipassing the make_plan. Find those places, check if the volatility check is needed (in most cases it would be safe to say yes) and try to use make_plan. This might not work though when async is not possible and/or envref is not available. Make a list of such cases. We need to discuss such cases individually.
-  The volatility of the recipe (e.g.in the get_asset) needs to be guaranteed. This should be a responsibility of the recipe provider. The recipe provider needs to check all the recipes for circularity and volatility. It should add a note to the recipe description "Recipe is VOLATILE due to ...". Similarly it should make a note about circular dependencies: "ERROR: Recipe has circular dependencies due to: ..." Recipe should have a flag has_circular_dependencies (by default false) and circular_dependency_key ... - that should prevent unnecessary re-evaluation of circularity.
-  in to_override - is there a reason why the existing cancel() method is not used?
-  A finished volatile asset should have a metadata with a) is_volatile = true and b) status:Volatile (not Ready). (This seems to be handled in Step 19, please check.)
-  In context::evaluate I realised it is a mistake - the evaluate should not make the assets volatile. It will be the so coalled side-effect assets (assets stored from a context in the asset manager and/or store) that will become volatile, but they are not implemented yet. Side-effect handling is out of scope. The is_volatile in context is a preparation to for the side-effect handling feature.
